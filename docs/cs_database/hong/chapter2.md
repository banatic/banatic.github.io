# Chapter 2 관계 데이터 모델과 제약조건

# 기본적인 용어

릴레이션(relation) >> 2차원 테이블

한 릴레이션은 데이터 베이스에 표현되는 한 엔티티에 관한 정보를 저장하는데 사용된다.

레코드(record) >> 릴레이션의 한 행(row)

공식적으로는 튜플(tuple)

릴레이션에 나타나는 엔티티의 특정 인스턴스에 관한 사실(값)들의 모임이다. 

릴레이션에 나타내는 엔티티의 한 인스턴스를 의미한다. 

애트리뷰트( attribute) >> 릴레이션에서 이름을 가진 하나의 열이다. 

도메인(domain) >> 한 애트리뷰트에 나타날 수 있는 값들의 집합

차수(degree) >> 한 릴레이션에 들어있는 애트리뷰트들의 수

유효한 릴레이션의 최소 차수는 1이다. 

카디날리티(cardinality) >> 릴레이션의 튜플 수 

널(NULL) >> 값이 없음, 알려지지 않음, 적용할 수 없음

릴레이션 스키마(relation schema) >> 릴레이션의 이름과 릴레이션의 애트리뷰트 집합

기본키에 밑줄을 귿는다. 

# 릴레이션의 특성

- 각 릴레이션은 오직 하나의 레코드 타입만 포함한다.
- 한 애트리뷰트 내의 값은 모두 같은 유형이다.
- 애트리뷰트들의 순서는 중요하지 않다.
- 릴레이션이 튜플들의 집합이기 때문에 동일한 튜플이 두 개 이상 존재하지 않는다.
- 한 튜플의 각 애트리뷰트는 원자 값을 가진다.(집합이 올 수 없다.)
- 릴레이션은 튜플들의 집합이기 때문에 튜플들의 순서는 중요하지 않다.
- 각 애트리뷰트의 이름은 한 릴레이션 내에서만 고유하다.

# 릴레이션의 키

1. 수퍼 키(super key)

    수퍼 키는 한 릴레이션 내의 특정 튜플을 고유하게 식별하는 하나의 애트리뷰트 혹은 애트리뷰트들 집합이다. 

    수퍼키는 튜플들을 고유하게 식별하는데 꼭 필요하지 않은 애트리뷰트들을 포함 할 수 있다.

2. 후보 키(candidate key)

    후보 키는 각 튜플을 고유하게 식별하는 최소한의 애트리뷰트들이 모임이다. 즉, 후보 키를 구성하는 애트리뷰트 중에서 어느 한 애트리뷰트라도 빼면 고유하게 식별하는 능력을 상실한다. 

    후보 키가 두개 이상의 애트리뷰트로 구성된다면 복합 키(composite key)라고 부른다. 

3. 기본 키(primary key)

    데이터 베이스 설계자가 여러개의 후보키 중에서 선택한 키

    종종 인위적인 키 애트리뷰트를 추가하게 되는데 이를 대리키(surrogate key)라고 한다. 

    기본 키는 널값이 될 수 없다.

     

4. 대체 키(alternate key)

    기본 키로 선정되지 않은 후보키

5. 왜래 키(foreign key)

    어떤 릴레이션의 기본 키를 참조하는 애트리뷰트이다. 

    외래 키는 관계 데이터베이스에서 릴레이션들 간의 관계를 나타내기 위해서 사용된다. 

    자기 자신을 참조할 수 도 있다.

# 무결성 제약조건

### 데이터 무결성(data integrity)

데이터의 정확성 또는 유효성을 

1. 도메인 제약조건(domain constraint)

    애트리뷰트 값은 반드시 원자 값이어야 하며, 데이터 형식을 통해 값들의 유형(정수형, 실수형, 문자형 등)  을 제한하고 애트리뷰트의 디폴트 값을 지정하고 애트리뷰트 값들의 범위를 지정할 수 있다.

2. 키 제약조건(key constraint)

    키 제약조건은 애트리뷰트에 중복된 값이 존재해서는 안된다는 것이다. 

3. 기본 키와 엔티티 무결성 제약 조건(entity integrity constraint)

    기본 키를 구성하는 애트리뷰트가 널값을 가지면 튜플들을 고유하게 식별할 수 없게 되므로 엔티티 무결성 제약 조건은 릴레이션의 기본 키를 구성하는 어떤 애트리뷰트도 널값을 가질 수 없다는 것이다. 

    어떤 엔티티에 관한 정보를 릴레이션에 저장하려면 식별이 가능해야 하므로 엔티티 무결성 제약조건이 필요하다. 

4. 외래 키와 참조 무결성 제약조건(referential integrity constraint)

    참조 무결성 제약조건은 두 릴레이션의 연관된 튜플들 사이의 일관성을 유지하는데 사용된다. 관계 데이터베이스에서 포인터 없이 관계를 릴레이션으로 표현한다. 따라서 관계도 하나의 릴레이션으로 나타내게 되고 한 릴레이션이 관계를 나타낼 때 다른 릴레이션의 기본 키를 "참조"하는 것을 기반으로 하기 때문에 외래키의 개념이 중요하고 참조 무결성이 지켜져야 한다. 

# 무결성 제약조건 유지

"참조하는" 릴레이션과 "참조 되는" 릴레이션 간의 참조 무결성 검증이 중요한 듯 하다. 

1. 삽입

    "참조되는" 릴레이션에 새로운 튜플이 추가되는 것은 무결성을 깨뜨릴 위험이 없다. 하지만 "참조하는"릴레이션에서의 삽입은 주의해야한다.

2. 삭제

    "참조하는" 릴레이션에 튜플이 삭제되는 것은 무결성을 깨뜨릴 위험이 없다. 하지만 "참조되는"릴레이션에서의 삭제는 주의해야 한다. 

    삭제에서 참조 무결성 제약조건을 만족시키기 위해서 DBMS는 몇가지 옵션을 제공한다. 

    1. 제한(restricted)

        위배를 야기한 연산을 단순히 거절한다.

    2. 연쇄(cascade)

        "참조되는" 튜플을 삭제하고 "참조하는" 릴레이션에서 삭제된 "참조되는"튜플의 키를 외래키로 가지고 있던 튜플을 삭제한다. 

    3. 널값(nullify)

        "참조되는" 튜플을 삭제하고 "참조하는" 릴레이션에서 삭제된 튜플을 참조하는 외래키 값에 NULL값을 넣는다. 

    4. 디폴트 값

        널 값을 넣는 대신에 디폴트 값을 넣는다. 디폴트 값이 명시되지 않았다면 연산을 거절한다. 

3. 수정

    수정하려고 하는 애트리뷰트가 기본 키인지 왜래키인지 검사한다. 수정하려는 애트리뷰트가 기본키, 왜래키 둘 다 아니라면 다른 제약조건을 만족하는 범위안에서 수정하면 된다. 

    기본 키나 왜래 키를 삭제하는 것은 위의 삭제 연산과 같다. (원래 있던 튜플을 삭제하고 새로운 튜플을 추가하는 것과 같다고 볼 수 있다.)
